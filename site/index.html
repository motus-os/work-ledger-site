Keep a local record of the commands that did the work.

Motus runs the command you already use, records its lifecycle in SQLite, and gives the run a stable ID and deterministic receipt. Arguments and raw output stay out of the ledger.

One command becomes one record.

Motus sits around a command, not inside your application. It preserves the small set of facts needed to find and inspect that run later.

  1. 01 Your command

    Run the script, test, build, or tool you already use.

    go test ./...
  2. 02 Motus wrap

    Motus starts it directly, forwards input and output, and records the lifecycle.

    motus wrap -- ...
  3. 03 Local ledger

    The run and its ordered events are committed to the project ledger.

    .motus/ledger.db
  4. 04 Receipt

    List the run, then project it into a compact JSON record.

    motus run receipt

Answer the practical questions later.

Use Motus when a command still matters after the terminal window or CI job is gone.

  • Which executable ran?Base name and argument count
  • Which source state did it use?Repository name, commit, and dirty state
  • When did it run?Start, end, and recorded timestamps
  • How did it finish?Outcome, exit code, signal, or timeout
  • How much output did it produce?Byte and newline counts by stream
  • Can I read the same record again?Stable run ID and deterministic receipt

Know what is in the ledger.

The default record is deliberately small. Your command still receives its arguments, input, environment, and output, but Motus does not copy their contents into SQLite.

Recorded

  • run and event IDs
  • start, end, and record timestamps
  • executable base name and argument count
  • Git repository name, commit, and dirty state
  • stdout and stderr byte and newline counts
  • exit code, signal, timeout, and outcome
  • generated lifecycle event payloads and hashes

Not recorded

  • argument values
  • stdin
  • raw stdout or stderr
  • environment variables or credentials
  • source files
  • prompts or transcripts
  • anything the command did outside the recorded lifecycle

A receipt tells you what the ledger recorded.

A closed run projects into compact JSON with ordered events and a receipt hash. Unchanged stored data produces the same receipt bytes each time.

{
  "schema": "motus.work-receipt.v1",
  "receipt": {
    "trust_model": "producer-controlled",
    "run": {
      "id": "run_7e2f...",
      "outcome": "success"
    }
  },
  "receipt_sha256": "06d315b0..."
}

Useful as a local process record

The receipt preserves the run fields and event order in a portable format. motus doctor checks the current schema, payload hashes, sequences, foreign keys, and terminal records.

Not independent proof

A person who controls the database can replace its controls and rewrite a self-consistent history. Motus does not sign the receipt, observe the work independently, or prove that the result was correct.

Before you use it.

Does Motus record every command?
No. You choose what matters by running it through motus wrap.
Why not use shell history or CI logs?
Use them when one system already answers the question. Motus helps when selected runs need one local index and one receipt shape.
Can I share a receipt?
Yes. It is portable JSON. A recipient should still treat it as producer-controlled unless a separate trusted system observed or attested to the work.
