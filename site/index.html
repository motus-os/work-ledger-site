One command becomes one record.

Motus sits around a command, not inside your application. It preserves the small set of facts needed to find and inspect that run later.

  1. 01 Your command

    Run the script, test, build, or tool you already use.

    go test ./...
  2. 02 Motus wrap

    Motus starts it directly, forwards input and output, and records the lifecycle.

    motus wrap -- ...
  3. 03 Local ledger

    The run and its ordered events are written to the project ledger.

    .motus/ledger.db
  4. 04 Receipt

    List the run, then project it into a compact JSON record.

    motus run receipt

Answer the practical questions later.

Use Motus when a command still matters after the terminal window or CI job is gone.

  • Which executable ran?Base name and argument count
  • Which Git context did it start from?Repository name, commit, and dirty state
  • When did it run?Start, end, and recorded timestamps
  • How did it finish?Outcome, exit code, signal, or timeout
  • How much output did it produce?Byte and newline counts by stream
  • Can I read the same record again?Stable run ID and deterministic receipt
  • Why did this run matter?A finding you chose to connect to it
  • What closed the finding?A dismissal note or a successful run connected as the fix

Keep the reason a run mattered.

A failed command gives you an outcome. A finding keeps the explanation or next step worth carrying forward, tied to that run.

$ ./bin/motus finding add \
    --run RUN_ID --file -
The generated file was stale.
(press Ctrl-D)
Recorded finding_91ac... (open)

$ ./bin/motus finding close finding_91ac... \
    --disposition resolved \
    --run FIX_RUN_ID --file -
Regenerated the file before the test.
(press Ctrl-D)
Closed finding_91ac... (resolved)

Output is condensed and IDs are shortened.

Connected to the origin run

Each finding keeps the stable ID of the closed run that gave it context.

Closed without rewriting it

A resolution names a closed successful run. A dismissal records a note instead. The original finding remains unchanged.

Search what you wrote

finding list --query stale searches the text you recorded and returns matching findings with their origin run IDs.

What Motus records.

Each wrapped run keeps its identity, Git context, timing, and outcome in the local ledger.

Run context

  • run and event IDs
  • start, end, and record timestamps
  • executable base name and argument count
  • Git repository name, commit, and dirty state

Run result

  • stdout and stderr byte and newline counts
  • exit code, signal, timeout, and outcome
  • ordered start and terminal records

A stable receipt for every closed run.

A closed run becomes compact JSON with ordered events and a receipt hash. Unchanged stored data produces the same bytes each time.

{
  "schema": "motus.work-receipt.v1",
  "receipt": {
    "trust_model": "producer-controlled",
    "run": {
      "id": "run_7e2f...",
      "outcome": "success"
    }
  },
  "receipt_sha256": "06d315b0..."
}

Portable and repeatable

The receipt preserves the run fields and event order in one portable format. motus doctor checks the current schema, payload hashes, sequences, foreign keys, terminal records, and finding links.

Findings stay separate from run receipts. Adding or closing one does not change the receipt for its run.

The receipt identifies its trust model as producer-controlled. Read what that means.

Where Motus fits.

When is a run worth recording?
When you may need its executable, Git context, timing, outcome, or finding after the terminal or CI job is gone.
What does Motus add beyond shell history or CI logs?
One local index, stable run IDs, searchable findings, and one receipt shape across selected work.
Can I share a receipt?
Yes. It is portable JSON with the recorded fields, event order, receipt hash, and stated trust model.
