Security

Motus protects a local ledger from accidental mutation and ordinary application writes. The database owner remains outside that boundary.

What Motus checks

  • state paths and private POSIX permissions
  • schema version and required SQLite objects
  • database integrity and foreign keys
  • event and finding payload hashes
  • event sequence order and finding run links
  • terminal events against their run projections

SQLite constraints and triggers reject ordinary event or finding mutation, deletion, sequence gaps, writes after a terminal event, run deletion, and changes to a closed run. A finding can be closed once without rewriting its original record.

Trust boundary

A person who controls the database file can drop the triggers, change records and hashes, and restore a self-consistent schema. A passing motus doctor report cannot prove this never happened.

Receipts are not signed. Motus does not independently observe the command or prove that every relevant event was captured, that the producer told the truth, or that the work was correct.

Sensitive data

Run lifecycle payloads are generated by Motus and exclude command argument values, stdin, raw output, environment variables, source files, prompts, and transcripts.

Finding text is different. Motus stores the summary, hypothesis, next step, or closure note that you explicitly submit through a file or standard input. Review that content before recording it.

Report a vulnerability

Do not open a public issue for a suspected vulnerability. Use the repository's private vulnerability-reporting form.

The complete, versioned policy lives in SECURITY.md.
